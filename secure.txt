---

> ## ⛔ ESTO NO ES UN TEST FUNCIONAL. ES UNA AUDITORÍA DE SEGURIDAD.
>
> Una auditoría de seguridad **no se limita a "probar como 4 tipos de usuario distintos".** Esa es una forma de pensar de tester funcional, no de auditor de seguridad, y produce auditorías superficiales que dejan pasar el 90% de las vulnerabilidades reales. Pensar solo en "deslogueado / usuario / admin logueado / admin deslogueado" es exactamente la trampa: asume que toda vulnerabilidad se manifiesta como diferencia de rol, y la mayoría no. Las vulnerabilidades viven en:
>
> - El código fuente, línea por línea — leído, no asumido
> - Race conditions entre requests concurrentes
> - Headers HTTP manipulados o ausentes
> - Deserialización de datos no confiables
> - Lógica de negocio explotable independientemente del rol (manipular cantidad/precio/IDs sin importar quién está logueado)
> - Cadenas de exploits: una vulnerabilidad de severidad baja combinada con otra de severidad baja que juntas son críticas
> - Configuración de infraestructura (Supabase, Next.js, variables de entorno, cabeceras, CORS, cookies)
> - Dependencias de terceros con CVEs conocidos
> - Lógica temporal (tokens que no expiran, ventanas de tiempo explotables, condiciones de carrera en checkout/inventario)
> - Errores de implementación que no tienen nada que ver con "estar logueado o no" — un bug de validación de tipo, un cast inseguro, una query mal parametrizada
>
> **Tu obligación es leer el 100% del código del proyecto, archivo por archivo, línea por línea, sin excepción**, y para cada pieza de lógica preguntarte activa y agresivamente: *¿cómo rompo esto?* No vas a organizar tu trabajo alrededor de "qué le pasa al usuario X" — vas a organizarlo alrededor de **qué hace cada archivo, cada función, cada query, cada endpoint, y de cuántas formas distintas se puede abusar de cada uno**, sin importar qué sesión tengas abierta en ese momento. Los estados de sesión (anónimo, autenticado, admin) son **una variable más a probar dentro de cada vector**, no la estructura organizativa de la auditoría.

---

> ## ⛔ LA ANALOGÍA DE LA PUERTA — Y POR QUÉ AQUÍ ES MÁS BRUTAL
>
> Si la chapa de una puerta no encaja, la solución no es quitar la chapa, ni la puerta, ni dejar el hueco vacío. Es desmontar toda la estructura — el marco, la pared, las bisagras — hasta entender por qué no encaja, y corregir exactamente esa pieza.
>
> En esta auditoría la analogía se extiende: **no basta con revisar si la puerta principal cierra bien.** Una auditoría enfermiza revisa **cada puerta, cada ventana, cada rejilla de ventilación, cada cable que entra y sale de la casa, los planos completos de la construcción, y el historial de cada remodelación que se le ha hecho.** Si encuentras una ventana sin pestillo, no das por buena la auditoría solo porque la puerta principal está bien. Revisas la casa entera, sin excluir nada porque "parece secundario" o porque "no está en la lista de las 4 puertas principales".
>
> **Cada vez que una corrección "funcione" porque algo dejó de validar, restringir o proteger, esa no es una solución — es quitar la chapa.** Descártala y sigue desmontando.
>
> **Esto aplica retroactivamente sin excepción.** Si durante tu lectura encuentras código que ya implementa un parche inseguro — RLS desactivado, una validación comentada, un `// TODO: arreglar esto después` que en realidad oculta un hueco de seguridad, un endpoint con autenticación condicionada por una variable de entorno mal pensada, lo que sea — lo tratas como un hallazgo más, lo diagnosticas de raíz, y lo corrigés sin reducir protección real.

---

> ## ⛔ PROHIBIDO INVENTAR. PROHIBIDO ASUMIR. PROHIBIDO USAR CHECKLISTS GENÉRICOS SIN VERIFICAR CONTRA EL CÓDIGO REAL
>
> No vas a aplicar una checklist de OWASP de memoria y marcar casillas sin evidencia. Cada ítem de esta auditoría se verifica **leyendo el código real del proyecto y el estado real de Supabase**, nunca por intuición ni por patrón genérico.
>

> 1. Lee el código completo relacionado, sin excepción

> 3. Confirma que tu solución es el patrón recomendado oficialmente para ese caso exacto, no un patrón genérico de "buenas prácticas" que recuerdas de otro contexto

---

> ## ⛔ EL ESTADO ACTUAL ES EL DISEÑO FINAL
>
> No cambias diseño visual. No cambias comportamiento funcional legítimo. Una corrección de seguridad nunca debe sentirse, para el uso legítimo del sistema, como pérdida de funcionalidad. Cierras lo que está expuesto sin tocar lo que protege bien ni lo que funciona bien para quien debe tener acceso.

## Fase 1 — Lectura forense del 100% del código (esto es el núcleo de la auditoría)

Esta fase no es opcional ni resumible. Aplica a **cada archivo** del inventario de la Fase 0.C.

### Protocolo de lectura forense por archivo

Para cada archivo, mientras lo lees línea por línea, evalúa explícitamente:

```
ARCHIVO: [ruta completa]
─────────────────────────────────────────────
¿Qué hace este archivo exactamente?
¿Recibe input de un usuario, de una API externa, o de la base de datos?
¿Ese input se valida? ¿Dónde exactamente y con qué reglas?
¿Ese input se usa para construir una query, una ruta de archivo, un comando, una URL?
¿Hay alguna verificación de autenticación antes de ejecutar lo que hace este archivo?
¿Hay alguna verificación de autorización (no solo "está logueado" sino "tiene permiso para ESTO específicamente")?
¿La verificación de autorización ocurre en el servidor, o solo se infiere en el cliente?
¿Este archivo expone, registra (log), o transmite algún dato sensible?
¿Este archivo usa alguna clave, token o secreto? ¿De dónde lo obtiene? ¿Está hardcodeado?
¿Este archivo tiene algún comentario tipo TODO, FIXME, HACK, XXX que sugiera una deuda de seguridad?
¿Qué pasa si este archivo recibe un input completamente inesperado (null, undefined, tipo incorrecto, extremadamente largo, con caracteres especiales)?
¿Qué pasa si dos requests a este mismo endpoint llegan al mismo tiempo (race condition)?
```

No avances al siguiente archivo sin completar este protocolo para el actual.

### Léelos en este orden, pero termina los ocho bloques sin excepción:

**Bloque 1 — Configuración y superficie de entrada:**
`package.json`, `.env*`, `next.config.*`, `tsconfig.json`, `middleware.ts`, cualquier archivo de configuración en la raíz, `vercel.json` o equivalente de despliegue si existe.

**Bloque 2 — Todo lo que toca autenticación y sesión:**
Cualquier archivo relacionado con login, registro, logout, manejo de tokens/cookies, helpers de Supabase Auth, cualquier wrapper o cliente de Supabase (server-side y client-side, **léelos por separado** — confirma cuál usa qué clave).

**Bloque 3 — Cada Route Handler / API Route / Server Action:**
Cada archivo bajo `app/api/`, `pages/api/`, o cada función marcada `'use server'`. Para cada uno: ¿qué verificación de auth/autorización tiene **al inicio de la función**, antes de tocar datos? ¿Qué pasa si se invoca directamente sin pasar por la UI?

**Bloque 4 — Cada query a Supabase en todo el proyecto:**

```bash
grep -rn "supabase\." --include="*.ts" --include="*.tsx" . | grep -v node_modules
```

Para cada resultado: ¿qué tabla toca? ¿es `select`, `insert`, `update`, `delete`, o `rpc`? ¿el filtro (`.eq()`, `.match()`, etc.) usa un valor que viene del cliente sin validar contra la sesión real del servidor? ¿se usa el cliente con `anon key` o con `service role key`? Si es Service Role, ¿por qué, y está justificado, y tiene su propio control de acceso en el código?

**Bloque 5 — Cada componente y página, sin excepción:**
Cada archivo en `app/`, `pages/`, `components/`. Aplica el protocolo forense completo a cada uno, prestando atención especial a: lógica de negocio que debería estar solo en el servidor pero está en el cliente (cálculo de precios, validación de stock, verificación de roles).

**Bloque 6 — Cada hook y cada util/helper:**
Todo en `hooks/`, `lib/`, `utils/`. Presta atención a funciones que parecen triviales pero manejan datos sensibles (formateo de precios que podría estar ocultando una validación faltante, parseo de JSON sin try/catch que podría crashear el servidor con un payload malformado).

**Bloque 7 — Tipos, esquemas de validación, y constantes:**
Cualquier archivo de tipos TypeScript, esquemas de validación (Zod, Yup, o similar si existen), y archivos de constantes. ¿Los tipos de TypeScript son la única validación, o hay validación real en runtime también? (TypeScript no protege en producción — solo en compilación.)

**Bloque 8 — Cualquier archivo no cubierto en los bloques anteriores:**
Estilos, configuración de íconos, archivos estáticos referenciados — verifica brevemente que ninguno contenga datos sensibles hardcodeados (claves, URLs internas, comentarios con credenciales de prueba reales).

## Fase 2 — Auditoría exhaustiva por categoría de vulnerabilidad

Con el código ya leído por completo, ejecuta cada una de estas categorías de forma activa y agresiva. Cada categoría se prueba **independientemente del estado de sesión** — pruébala sin sesión, con sesión regular, y con sesión admin como variantes dentro de la misma categoría, no como ejes separados de la auditoría.

Para cada hallazgo:

```
HALLAZGO: [identificador]
CATEGORÍA: [ver lista abajo]
Archivo(s) involucrado(s): [ruta exacta]
Vector exacto:     [pasos exactos para reproducir, incluyendo qué sesión si aplica]
Resultado esperado (comportamiento seguro): [qué debería pasar]
Resultado real observado: [qué pasó]
¿Es vulnerabilidad?: [SÍ/NO — justifica]
Severidad: [CRÍTICA / ALTA / MEDIA / BAJA / INFORMATIVA]
Evidencia: [código, request/response, query SQL ejecutada, log]
Impacto si se explota: [qué puede hacer un atacante real con esto]
```

### 2.A — Inyección (SQL, NoSQL, comandos, headers)

- [ ] Cada query a Supabase: ¿usa siempre el query builder parametrizado, o hay algún punto con concatenación manual de strings hacia SQL crudo (`rpc` con SQL dinámico, por ejemplo)?
- [ ] Cada función de Supabase (`Database Functions`) marcada `SECURITY DEFINER`: ¿valida correctamente sus parámetros? ¿puede ser invocada para hacer algo no previsto?
- [ ] Header injection: ¿algún valor de input del usuario se usa para construir headers de respuesta o redirects sin sanitizar?
- [ ] Inyección de comandos: ¿hay alguna llamada a `exec`, `child_process`, o similar en el código del servidor que use input de usuario?

### 2.B — Autenticación (probada de forma agresiva, no superficial)

- [ ] Fuerza bruta sin rate limiting: automatiza (conceptualmente, documenta el intento) múltiples intentos de login fallidos consecutivos. ¿En qué punto el sistema reacciona, si reacciona?
- [ ] Enumeración de usuarios vía mensajes de error distintos entre "usuario no existe" y "contraseña incorrecta".
- [ ] Enumeración de usuarios vía el flujo de registro (¿el sistema revela si un email ya está registrado de forma que permita enumerar la base de usuarios?).
- [ ] Session fixation: ¿el ID de sesión/token cambia después de un login exitoso, o se reutiliza el que existía antes de autenticarse?
- [ ] Tokens JWT (si se inspeccionan directamente): ¿qué algoritmo usan? ¿se valida la firma siempre? ¿hay información sensible en el payload del JWT que sea legible sin descifrar (recuerda que un JWT no es cifrado, solo firmado)?
- [ ] ¿El logout invalida la sesión en el servidor (revoca el refresh token), o solo borra el token del lado del cliente dejando la sesión viva en Supabase?
- [ ] Reutilización de tokens después de logout: guarda un token antes de cerrar sesión y verifica si sigue siendo válido después.

### 2.C — Autorización, IDOR, y escalación — probado en cada endpoint y cada componente, no en "4 estados"

- [ ] Para **cada Route Handler / Server Action identificado en la Fase 1, Bloque 3**: invócalo directamente (sin pasar por la UI) variando: sin sesión, con sesión regular, con sesión admin, y con un `id` o recurso que no pertenece a la sesión actual. Documenta el resultado de cada combinación para cada endpoint, no solo para los "endpoints de admin obvios".
- [ ] IDOR sistemático: para cada recurso identificable por ID (producto, orden, usuario, dirección, favorito, carrito) prueba acceder/modificar/eliminar un ID que no corresponde a la sesión actual.
- [ ] Mass assignment: en cada formulario/endpoint de creación o edición, intenta enviar campos adicionales no esperados por la UI (por ejemplo, un campo `role: "admin"` o `isAdmin: true` en el payload de registro o edición de perfil) y verifica si el servidor los ignora o los acepta.
- [ ] Verifica que el rol del usuario (admin/regular) se determine **siempre** consultando la fuente de verdad en el servidor (base de datos o claim verificado del JWT), nunca desde un valor enviado por el cliente en el body, query string, o header.
- [ ] Bypass de middleware: si existe middleware de protección de rutas, confirma que el Route Handler subyacente también valida por sí mismo — el middleware nunca debe ser la única capa de protección.

### 2.D — RLS de Supabase — tabla por tabla, exhaustivo, vía MCP/SQL Editor

Para **cada tabla** sin excepción:

- [ ] ¿RLS habilitado? Si no, en una tabla no estrictamente pública: **hallazgo crítico**.
- [ ] Lista cada política (`SELECT`/`INSERT`/`UPDATE`/`DELETE`) con su `USING`/`WITH CHECK` exacto.
- [ ] Ejecuta queries reales vía SQL Editor o MCP simulando rol anónimo, un `auth.uid()` de usuario regular, y un `auth.uid()` de admin, contra cada tipo de operación en cada tabla. Compara el resultado real contra lo que debería pasar.
- [ ] ¿Alguna política usa `USING (true)` o equivalente en una tabla que no debería ser totalmente pública para esa operación?
- [ ] ¿La determinación de "es admin" dentro de una política RLS se basa en una fuente confiable (tabla de roles, claim verificado), o en algo manipulable?
- [ ] ¿Hay vistas (`views`) o funciones que expongan datos sin pasar por las políticas RLS de la tabla base?

### 2.E — Claves y secretos

- [ ] `grep` de cualquier patrón de clave o secreto en todo el código fuente, no solo donde "se espera":

```bash
grep -rniE "(service_role|secret|api[_-]?key|password|private[_-]?key)" --include="*.ts" --include="*.tsx" --include="*.js" . | grep -v node_modules
```

- [ ] Confirma que `SUPABASE_SERVICE_ROLE_KEY` nunca llega al bundle del cliente.
- [ ] Revisa el historial completo de git por secretos commiteados alguna vez, aunque después se hayan removido:

```bash
git log --all --full-history -p | grep -niE "(service_role|secret|api[_-]?key)" 
```

- [ ] ¿Cada Route Handler que usa Service Role tiene su propio control de acceso explícito en el código, dado que Service Role bypassa RLS por completo?

### 2.F — Storage

- [ ] Buckets públicos vs privados: ¿la configuración corresponde al contenido real almacenado?
- [ ] Policies de Storage: ¿permiten subida sin restricción de tipo/tamaño? ¿permiten sobreescribir archivos ajenos?
- [ ] Acceso directo a archivos de buckets privados sin sesión o con sesión de otro usuario.
- [ ] Subida de archivos con extensión peligrosa (`.html`, `.svg` con script embebido) donde se espera solo imagen — y verifica si ese archivo, una vez servido, se interpreta como HTML/JS por el navegador (Content-Type incorrecto).

### 2.G — Lógica de negocio explotable (independiente de rol)

- [ ] Precio: ¿se recalcula siempre en el servidor en cualquier operación que mueva dinero (agregar al carrito, checkout), o el cliente puede enviar un precio arbitrario? Pruébalo manipulando el payload directamente con Edit and Resend.
- [ ] Cantidad/stock: ¿se valida en servidor? ¿es posible comprar/reservar más unidades de las que hay en stock mediante requests concurrentes (race condition de inventario)?
- [ ] Promociones/descuentos: ¿es posible aplicar un código o estado de promoción expirado, inactivo, o no propio manipulando el request?
- [ ] Productos ocultos/inactivos/borrador: ¿son accesibles manipulando directamente su ID en una request, saltándose el filtro de visibilidad que aplica la UI?
- [ ] Operaciones idempotentes: ¿qué pasa si el mismo request de "agregar al carrito" o "crear orden" se envía dos veces rápidamente (doble click, replay de request)? ¿se duplica indebidamente?

### 2.H — Inputs, XSS, deserialización

- [ ] Cada campo de texto libre probado con payloads XSS (`<script>alert(1)</script>`, `<img src=x onerror=alert(1)>`, atributos con `javascript:`). ¿Algún punto lo renderiza sin escapar?
- [ ] Cada uso de `dangerouslySetInnerHTML`: ¿de dónde viene el contenido? ¿está sanitizado con una librería real (DOMPurify o similar) o solo "se ve que viene de admin así que es confiable"?
- [ ] Parseo de JSON proveniente de input externo: ¿está envuelto en try/catch? ¿qué pasa si llega un JSON malformado o un payload extremadamente anidado/grande?
- [ ] Validación de tipo en el servidor para cada campo numérico (precio, cantidad, stock) — no confíes en que el cliente solo envía números.

### 2.I — Cabeceras, cookies, configuración de transporte

- [ ] Cabeceras de respuesta presentes/ausentes: `Content-Security-Policy`, `X-Frame-Options`, `Strict-Transport-Security`, `X-Content-Type-Options`, `Referrer-Policy`.
- [ ] Cookies de sesión: flags `HttpOnly`, `Secure`, `SameSite` correctos.
- [ ] Clickjacking: ¿el sitio puede embeberse en un iframe de otro origen?
- [ ] CORS: revisa la configuración — ¿permite orígenes arbitrarios (`*`) en endpoints que manejan datos sensibles o autenticados?
- [ ] Mensajes de error visibles al usuario final: ¿exponen stack traces, rutas de archivos del servidor, nombres de tablas o columnas, versiones de librerías?

### 2.J — Dependencias y configuración de despliegue

- [ ] Resultado completo de `npm audit` (de la Fase 0.A) — cada hallazgo `moderate` o superior evaluado contra el uso real en el código.
- [ ] `next.config.*`: revisa `images.domains`/`remotePatterns` (¿permite cargar imágenes de cualquier dominio, habilitando SSRF a través del optimizador de imágenes de Next.js?), headers personalizados, redirects/rewrites que puedan exponer rutas no previstas.
- [ ] Variables de entorno: cada una documentada, cada una con el prefijo correcto (`NEXT_PUBLIC_` solo donde debe ser pública).

### 2.K — Cadenas de exploits

Esta categoría es exclusiva de una auditoría seria: revisa si **dos o más hallazgos de severidad baja o media, combinados, producen un impacto crítico.** Ejemplos del tipo de razonamiento que debes aplicar (no asumas que estos existen, evalúalo contra los hallazgos reales que encontraste): un XSS de severidad media combinado con una cookie de sesión sin `HttpOnly` puede convertirse en robo de sesión completo. Un IDOR de severidad media que expone el email de un usuario combinado con un flujo de recuperación de contraseña débil puede convertirse en toma de cuenta completa. Documenta explícitamente si encontraste alguna combinación de este tipo.

Consolida todo lo de la Fase 2 en un único registro:

```
SEC-001
  Categoría:        [2.A a 2.K]
  Archivo(s):       [ruta exacta]
  Vector exacto:    [pasos exactos para reproducir]
  Severidad:        [CRÍTICA / ALTA / MEDIA / BAJA / INFORMATIVA]
  Evidencia:        [código / request-response / SQL ejecutado / log]
  Impacto:          [qué puede hacer un atacante real]
```

Criterio de severidad:

- **CRÍTICA:** acceso no autorizado a datos ajenos, escalación a admin, RLS deshabilitado en tabla sensible, Service Role expuesta, bypass total de auth, RCE, SSRF con acceso a infraestructura interna.
- **ALTA:** IDOR confirmado, autorización solo en cliente sin respaldo en servidor, manipulación de precio/stock aceptada por el servidor, XSS persistente.
- **MEDIA:** falta de rate limiting, mensajes de error que filtran información, cabeceras ausentes, validación de input incompleta sin explotación directa demostrada aún, dependencias vulnerables sin uso confirmado de la parte afectada.
- **BAJA/INFORMATIVA:** buenas prácticas ausentes sin vector de explotación directo y claro en este momento.

No omitas ningún hallazgo por parecer menor — la severidad prioriza el orden de corrección, no decide qué se documenta. Documenta también explícitamente las cadenas de exploits de 2.K como hallazgos propios con su severidad combinada.

## Fase 4 — Diagnóstico bajo la analogía de la puerta (por cada hallazgo, en orden de severidad)

### Paso 1 — Desmonta toda la estructura

1. Lee el archivo completo donde está la lógica vulnerable.
2. Todos los archivos que importa y todos los que lo importan:

```bash
grep -r "from.*[nombre-del-archivo]" --include="*.ts" --include="*.tsx" --include="*.js" --include="*.jsx" . | grep -v node_modules
```

3. La política RLS real involucrada, verificada en vivo (MCP o Dashboard).
4. El flujo completo cliente → servidor → base de datos.
5. **Todos** los puntos de entrada a esa misma operación (UI, API directa, Server Action, posible acceso vía Storage o RPC) — una corrección que cubra solo uno de los caminos no es completa.

### Paso 2 — Documentación oficial antes de proponer nada

Identifica la tecnología exacta involucrada y consulta su documentación oficial de seguridad para el caso específico. Confirma que existe un patrón oficial recomendado para este escenario exacto.

### Paso 3 — Causa raíz

```
CAUSA RAÍZ:
  Archivo:                   [ruta exacta]
  Línea(s):                  [número]
  Código exacto:             [fragmento]
```
